6 Ways to Deploy a Simple App on AWS

Posted by Saravanesh on June 7, 2025

“Simplicity is the ultimate sophistication.”

Real-world apps often look complicated — but that’s only because we’re seeing the end result. What starts as a simple setup grows over time, one feature at a time. Build smart, layer slowly, and before you know it, your app looks "complex" to everyone else.

Thanks to AWS, deploying a basic app doesn’t need a battalion of engineers. You can go from zero to live in under an hour. Whether you want total control or prefer to let AWS handle the heavy lifting, there’s a way for you.

Let’s look at six practical ways to deploy a simple app on AWS — pick what fits your style and scale up as you grow.

1. EC2 – Do It Yourself (Classic Method)

This traditional approach gives you a virtual Linux server in the cloud where you can install and run any type of application manually. For web apps, you can use Nginx or Apache to serve static or dynamic content; for APIs, you might go with Node.js + PM2, or Python3 + Gunicorn + Flask/Django; and for PHP-based applications like WordPress, Apache with PHP-FPM works well.

You can also set up databases like PostgreSQL, MySQL, or MongoDB, and use Redis for caching. If you want CI/CD or version control, tools like Jenkins and Docker can be installed. You can even secure your server with Fail2ban, UFW, and Certbot for SSL.

This method gives you complete control, ideal for learning cloud infrastructure, working with open-source tools, and building custom setups—but it also means you manage updates, patches, and security manually. Use it if you want flexibility and hands-on experience with the Linux ecosystem on the cloud.

2. EC2 with Launch Template + User Data – Semi-Automated Setup

This is an improved version of the EC2 method. Instead of manually logging into each instance and setting it up every time, you can create a template that includes the instance details and a script to automatically install and configure your app when it boots.

For example, you can write a script that updates the OS, installs a web server, and writes your app’s homepage. Every time you launch an instance from this template, the setup runs automatically.

This is ideal when you need to create the same environment repeatedly, especially for testing or training purposes.

3. ECS with Fargate – Serverless Containers

Even though this method requires your application to be containerized, containerizing is now a common and efficient way to package any kind of app—be it a web server, API service, background processor, or even a database—for portability and consistency across environments.

Once your app is in a Docker image, ECS with Fargate makes it easy to run without managing any underlying servers. You simply define the container requirements—like CPU, memory, ports, and environment variables—and AWS handles the rest: provisioning, patching, scaling, and high availability. You upload your Docker image to Amazon ECR (Elastic Container Registry), configure a task definition, and deploy. It’s ideal for microservices, APIs, or any app where you want clean separation, easy scaling, and minimal infrastructure overhead.

This method is especially useful in production environments or when you're aiming for consistent, reproducible deployments with less operational friction.

4. Lambda with API Gateway – Pure Serverless

For lightweight applications or quick backend logic that just needs to respond to events—like processing a contact form, sending an email, resizing an image, or fetching data on request—AWS Lambda is your minimalist powerhouse.

You write the function code, upload it (or write inline), and AWS executes it only when triggered. To turn it into a fully functional web API, you hook it up with API Gateway, which gives you REST or HTTP endpoints instantly.

For example, you could build a feedback form handler where the frontend sends form data to an API Gateway endpoint, which triggers a Lambda to validate and store the data in DynamoDB—no servers, no idle costs.

This combo is perfect for event-driven utilities, micro-backends, or adding modular logic into a larger system. It’s the kind of approach that scales invisibly and keeps your infrastructure footprint almost nonexistent.

5. AWS Lightsail – Simplicity First

Lightsail is AWS’s beginner-friendly service. It offers pre-configured virtual machines where you can launch common stacks like WordPress, LAMP, Node.js, or even plain Linux with a few clicks.

You don’t need to know about VPCs, subnets, or IAM roles. Just choose a blueprint, pick a plan (with predictable pricing), and launch. It’s like a simplified version of EC2 with built-in networking, firewall, and snapshots.

Perfect for small websites, personal blogs, or early MVPs where speed and simplicity matter more than deep configuration.

6. AWS Amplify – Fullstack for Frontend Developers

If you’re building a frontend app using React, Vue, or Angular, Amplify offers an all-in-one platform. You can connect your code repository (like GitHub), and Amplify will automatically build, host, and deploy your app every time you push changes.

It also provides easy ways to add backend features like authentication, APIs, and storage — all through a CLI tool or visual interface. You don’t need to write backend code unless you want to.

This is excellent for solo developers or small teams who want to ship fast without worrying about servers or backend complexity.

Final Thoughts

As you can see, there isn’t just one way to deploy a web app on AWS. Each method serves a purpose. Start simple, understand the tools, and evolve based on your app's complexity and scale. Whether you're someone who likes to build everything from scratch or prefers a few clicks and some magic — AWS supports your flow.

Even when you're building a small app for practice, it's worth treating it like a real project. Separating code, using Docker, writing a few tests, and adding basic logging not only sharpens your skills but also sets a foundation for scalability. Document everything and design your app as if a team might join you tomorrow. This approach turns tiny projects into portfolio pieces or startup ideas. With the right mindset, tomorrow, it could first Impress a recruiter, then land a client, become a product, and finally turn you into a founder.

Practice like you're building the future — because maybe, you are.

The Battle Against AWS Charges

Posted by Saravanesh on March 21, 2025

For many developers, startups, and learners, AWS Free Tier is an excellent way to explore cloud computing without costing anything. It allows users to experiment with Cloud services and build small projects with free resources. However, what starts as an exciting cloud journey often turns into an unexpected billing nightmare.

Many users have faced the shock of seeing unexpected charges appear in their billing dashboard, even when they assumed they were well within Free Tier limits. This is because of some unattended resources quietly accumulating costs. Some common culprits in AWS include:

  • EC2 instances running beyond their intended time, consuming compute hours.
  • Unattached EBS volumes that still incur storage charges.
  • Idle RDS databases silently adding to the monthly bill.
  • Unassociated Elastic IPs sitting idle but generating costs.

Forgetting to manually delete these resources is easy, and cloud providers doesn't provide a built-in mechanism to automatically clean up unused resources. So, how can we ensure that we stay within Free Tier limits without constant manual monitoring?

I’ve put together a straightforward solution using AWS Lambda and EventBridge to automate the cleanup process, keeping your cloud environment optimized, preventing unexpected charges, and saving time. In this guide, we'll walk through how to set up an effective Free Tier cleanup automation, ensuring no unnecessary costs catch you off guard. With some tweaks, this approach can also be adapted for other cloud platforms as well.

Also note that AWS Free Tier includes 1 million Lambda requests and 400,000 GB-seconds of compute time per month, so running this cleanup function every 30 minutes will stay well within the free limits. Similarly, Amazon EventBridge offers 1 million free events per month, ensuring that this automation won’t add any unexpected costs.

This means you can set up automated cleanups without worrying about charges, keeping your cloud resources optimized while fully leveraging the AWS Free Tier.

Automating Free Tier Cleanup

  1. Terminate EC2 instances running over 30 minutes – Preventing unnecessary usage.
  2. Delete unused EBS volumes – Freeing up space and avoiding storage costs.
  3. Release unassociated Elastic IPs – Ensuring no charges for unused resources.
  4. Stop idle RDS databases – Avoiding excess charges from inactive databases.
  5. Delete old RDS snapshots – Managing storage efficiently.

GitHub Repository: AWS Free Tier Clean Up

Once these scripts are ready, scheduling them to run automatically ensures no manual intervention is required.

Scheduling with EventBridge

AWS EventBridge can be used to automate cleanup tasks:

  1. Create a rule named aws-free-tier-cleanup.
  2. Set a schedule expression: cron(0 0,6,12,18 * * ? *) to run every 6 hours.
  3. Choose Lambda Function as the target.
  4. Link it to the respective cleanup functions.
  5. Click Create.

The Outcome

With automation in place, unexpected AWS charges become less of a concern. Regular cleanups help maintain an optimized environment within Free Tier limits.

To further enhance cost efficiency:

  • Extend cleanup scripts to manage S3 storage and CloudWatch logs.
  • Send alerts via AWS SNS when resources are deleted.
  • Deploy everything with Terraform for seamless infrastructure management.

By implementing these strategies, AWS costs remain predictable and under control, ensuring a smooth experience without billing surprises. The entire setup requires minimal effort and provides long-term savings, making it a valuable addition to any AWS environment.

Understanding System Architecture Through Human Emotions

Posted by Saravanesh on November 25, 2024

When I was a developer, it felt like I was solving puzzles in isolation, unaware of how they fit into the bigger picture. I’d code, test, and deliver without fully grasping how my work impacted the lives of people who used the software. As years passed, I began to see the patterns—like pieces of a story coming together. I realized that understanding the purpose and business need of an application is as crucial as writing the code itself. And what better way to explain this than by drawing parallels with something deeply relatable: human emotions and relationships?

Just as humans are a textile of emotions working in harmony, system architecture is a collection of interconnected components working together to create seamless experiences. Let’s break this down, one emotion at a time.

1. Frontend: The Smile You Show the World

Think of the frontend as a person’s expression—the smile, the gestures, the tone of voice. It’s what people see first and how they judge your mood and intentions. For a system, the frontend is the website or app interface. It’s the friendly face of your system, guiding users to what they need while making them feel comfortable and welcome.

When a grocery store app makes it easy to browse and shop, it’s like a friend with a warm smile who says, “I’m here for you. Let me help.”

2. Backend: The Heart of the Matter

The backend is the emotional core—the place where thoughts, decisions, and processes originate. Just as your emotions guide your actions, the backend drives the logic of an application. It processes your requests, ensures everything runs smoothly, and communicates with other systems.

Imagine placing an order on your grocery app. The backend works like the emotions behind the scenes, ensuring the right responses are triggered: “They’re hungry? Let’s make sure the order is prepared on time.”

3. Database: The Memory That Holds It All

Memories shape who we are. Similarly, the database is the repository of all past experiences for a system—storing customer details, product inventories, and order histories. It’s like the collective memory that helps us learn, grow, and adapt.

When you re-order your favorite snack from the grocery store, the database recalls your preferences, like a friend who remembers your coffee order without asking.

4. APIs: The Conversations We Have

APIs are the conversations between systems, just as communication is the lifeline of relationships. They ensure everyone is on the same page, connecting payment gateways, delivery services, and tracking systems.

It’s like calling a friend to confirm dinner plans: “Are you ready? Should I pick you up?” APIs keep things moving smoothly, ensuring all parts work in harmony.

5. Load Balancer: Managing Stress with Grace

Life throws challenges at us, and how we manage stress determines our resilience. The load balancer is like emotional regulation—it ensures no part of the system is overwhelmed.

Imagine a busy grocery app during festive seasons. The load balancer ensures every server shares the load, much like how a calm mind helps you juggle multiple tasks without losing focus.

6. Cloud: The Capacity to Grow and Adapt

Our capacity to grow, adapt, and expand our horizons mirrors the cloud. Just as we find strength during tough times or grow when faced with challenges, the cloud scales the system to handle increasing demands.

During a sale, when thousands of people log in to shop, the cloud expands resources, much like how we tap into reserves of patience and strength when life demands more from us.

7. Security: The Guardrails of Trust

Trust is the foundation of any relationship, just as security is the backbone of any system. It protects personal data, payment details, and sensitive information, ensuring users feel safe and cared for.

Think of it as a protective instinct, like when a parent ensures their child crosses the road safely. Security creates a safe environment for users to engage without worry.

8. DevOps: The Caretaker of Relationships

DevOps is like the invisible support system in relationships—the person who keeps things running, fixes issues, and ensures the connection remains strong. It’s the team that releases updates, fixes bugs, and enhances features without disrupting the user experience.

In our grocery store analogy, DevOps ensures that the app is always improving—adding faster delivery options or personalizing your experience, just like how effort keeps a relationship alive and thriving.

9. The Big Picture: An Online Grocery Store and Human Life

Bringing it all together is like seeing the connection of human emotions. Every system component works like an emotion or action in a human being, creating a harmonious experience.

Let’s take an online grocery store as an example:

  • Frontend: The smile that greets you when you open the app.
  • Backend: The heart that processes your needs and makes decisions.
  • Database: The memory that recalls your favorite items.
  • APIs: The conversations that coordinate between payment, delivery, and tracking.
  • Load Balancer: The calm under pressure during a rush hour.
  • Cloud: The ability to adapt and grow during busy times.
  • Security: The trust that ensures your data is safe.
  • DevOps: The ongoing effort to keep improving and maintaining the relationship.

Conclusion

As developers, when we look beyond the lines of code, we see the bigger picture. Just as emotions make us human, system architecture brings life to applications. Understanding the purpose behind every layer helps us build better systems and create a more profound impact in the real world.

So, next time you design or build something, think about the human emotions it mirrors. The more empathy you bring to your code, the better your creations will resonate with the people who use them.

How GenAI Could Transform Our Language with Database

Posted by Saravanesh on November 12, 2024

The languages we use to communicate—whether with each other or with computers—have always evolved, moving from rigid codes to intuitive, natural interactions. Just as spoken languages change, computer languages are poised to do the same. One of the biggest transformations could be the way we retrieve data: a future where SQL and its structured commands give way to fluid, conversational AI-driven queries.

Imagine asking a database, “What were last quarter’s sales?” and receiving instant, relevant insights without needing to know SQL syntax. With GenAI, we could interact with data naturally, bridging the gap between rigid database commands and flexible, accessible querying.

Replacing SQL with Natural Language

Traditionally, accessing data in SQL requires knowing technical details—table names, fields, and conditions—to build queries. GenAI, however, could interpret plain language requests, making data retrieval as simple as asking a question. Developers and business users alike would benefit from a system that adapts to their language, eliminating the need to learn or remember specific syntax. This approach would make the data itself more accessible, empowering everyone to make data-informed decisions.

Unified Access to All Data Types

One of the most exciting capabilities GenAI could unlock is the unification of all data types—structured, semi-structured, and unstructured—under one querying system. Instead of enforcing rigid tables and schemas, AI could index data on-the-fly, allowing retrieval from various sources: structured databases, customer emails, sales reports, even multimedia files. This ability to cross-query without restructuring could bring previously siloed information together, making insights more immediate and relevant.

Visual and Interactive Data Manipulation

Imagine a future where querying data doesn’t require code but works through an interactive, visual interface. Dragging and dropping fields or highlighting areas of interest, users could explore data instantly, with GenAI recalculating and refining results as they go. This level of interaction would make working with data feel as simple as sorting through a file cabinet—only vastly faster and smarter.

Databases Optimized for AI

In the future, databases could be designed with AI in mind, shifting from rigid structures to more adaptive formats. Instead of enforcing strict schemas, databases could evolve to allow on-the-fly indexing and dynamic organization. This adaptability would enable natural language questions to pull data directly, making retrieval not only faster but more flexible and accessible.

Vector Searching: The Next Frontier in Data Querying

In addition to natural language querying, another powerful concept that could revolutionize how we retrieve data is vector searching. Traditional relational databases rely on structured queries like SQL to fetch exact matches from tables, whereas vector searching leverages machine learning to understand the meaning behind the data.

Vector databases use embeddings—high-dimensional representations of data—to perform similarity searches. Instead of matching exact keywords, a vector search can find related data based on context. For example, a query for "best-selling products in Q1" could return related results based on customer preferences, trends, or other patterns that might not exactly match the query but are contextually relevant.

Challenges and the Path Ahead

This shift from SQL to GenAI-driven querying won’t be without its challenges. Ensuring accuracy, context, and security in AI-driven data retrieval will require advancements in AI accuracy and access control. But if these hurdles can be overcome, GenAI could make data access universally available and intuitive.

As I experiment with building this kind of GenAI-enabled tool, I’m exploring how it can unify data, simplify access, and make retrieval secure—all while eliminating the language barrier between users and databases. This could redefine how we interact with data at the developer level and beyond, and I’ll be sharing more of my insights in upcoming blog posts.

Synchronous vs. Asynchronous Operations

Posted by Saravanesh on August 20, 2024

In computing, the concepts of synchronous and asynchronous operations often come up. But these ideas aren’t limited to code; they exist in everyday experiences too. Take, for example, a recent visit to two different cinemas.

At PVR cinemas, the process was straightforward. I placed my order at the counter, and the attendant completed it before moving on to the next customer. This is a textbook example of a synchronous operation. Each step follows the previous one, and nothing else happens until the current task is finished. It’s efficient in its own way but requires patience.

In contrast, at another cinema, I placed my order at one counter and collected it at another. This approach exemplifies asynchronous operations. Multiple tasks were handled simultaneously: while my payment was being processed, someone else was preparing my order. The result? Faster service with tasks running in parallel, although slightly less structured.

These two experiences highlight the fundamental difference between synchronous and asynchronous operations. In a synchronous system, tasks are completed one after another, ensuring that each step is fully executed before the next begins. This method is useful when order and sequence are crucial. Asynchronous systems, on the other hand, allow tasks to occur concurrently, improving overall speed but introducing complexity in managing those tasks.

Understanding these concepts is key in both programming and system design, whether you’re optimizing a codebase or simply waiting in line for popcorn.

Building a Two Way Data Sync

Posted by Saravanesh on April 4, 2024

The synchronization of data between AWS and on-premises systems represents a pivotal challenge in today's hybrid computing landscape. My exposure to this intricate process came through practical implementation within my professional environment, shedding light on its complexities and nuances.

When the system was migrated to the cloud, I faced the challenge of learning a new technology stack. I familiarized myself with cloud computing concepts such as virtualization, containerization, and serverless computing. I also learned about specific cloud services such as Amazon Elastic Compute Cloud (EC2), Amazon Simple Storage Service (S3), and Amazon Lambda.

I’ve been able to navigate the intricacies of AWS and database management, albeit with a learning curve. The realization dawned upon me that in the vast expanse of cloud computing and data management, there's always more to explore and comprehend.

Central to our synchronization strategy is Amazon Database Migration Service (DMS), a robust tool facilitating seamless data replication across disparate environments. DMS operates as the linchpin of our setup, diligently capturing changes from our on-premises Postgres SQL database and mirroring them in near real-time on AWS. This ensures that our data remains synchronized across both platforms, vital for our operational continuity and decision-making processes.

Complementing DMS is the implementation of a temporary schema within our Postgres SQL database. This schema acts as a staging area, enabling the aggregation and validation of incoming data changes before they are integrated into the main database. This approach not only safeguards data integrity but also streamlines the synchronization process, minimizing disruptions and maximizing efficiency.

To orchestrate the synchronization tasks seamlessly, we rely on cron job schedulers. These schedulers are configured to trigger synchronization operations at predetermined intervals, ensuring that data updates are propagated swiftly and without human intervention. This automation not only enhances operational efficiency but also reduces the risk of human error, critical in maintaining data consistency across environments.

While our synchronization methodology proves effective, it's not without its challenges, notably in terms of cost management. As data volumes grow and synchronization demands increase, the associated costs can escalate rapidly. To mitigate this, we're exploring various cost optimization strategies.

These strategies include optimizing resource utilization, leveraging open-source solutions, implementing change data capture mechanisms, exploring hybrid storage options, and utilizing AWS Spot Instances and Spot Fleets. By embracing these approaches, we aim to strike a balance between cost-effectiveness and operational excellence, ensuring that our data synchronization efforts remain sustainable and scalable.

In conclusion, the synchronization of data between AWS and on-premises systems is a multifaceted endeavor, demanding a nuanced approach and ongoing refinement. Through the integration of Amazon DMS, Postgres SQL, and meticulous cost optimization strategies, we're able to achieve near real-time data synchronization while navigating the complexities of the cloud computing landscape.

In this blog post, we'll explore the key concepts of AWS and how it revolutionizes the way we deploy and manage applications in the cloud, along with my personal experience in Tech industry.

From virtual servers (EC2) to managed databases (RDS) and content delivery networks (CloudFront), AWS provides a rich set of tools that empower developers and businesses to innovate and scale.
